# Blog Architecture

A client-side SPA for browsing hierarchical markdown documentation, deployed as static files on Cloudflare Workers. No bundler, no framework — Alpine.js for reactivity, marked.js for markdown parsing, KaTeX for math, Highlight.js for code.

## Module Map

```
index.html               → entry point, loads scripts in order
├── utils.js             → slug generation (shared: router, TOC, search)
├── navigation-data.js   → static page tree (auto-generated by scripts/gen-nav.js)
├── marked-extensions.js → protects LaTeX delimiters ($$ / $) from marked parsing
├── router.js            → History API routing; validates paths, fires onRouteChange
├── search.js            → full-text search, indexed at startup
├── content-loader.js    → fetches markdown, rewrites links, caches parsed HTML
├── toc-generator.js     → extracts h2/h3, builds TOC, tracks active section
├── renderer.js          → KaTeX, Highlight.js + toolbar, timeline class detection
└── app.js               → Alpine.js app state, wires all modules together
```

Scripts load in order via `<script>` tags. Each module is a plain class; `app.js` initializes them after the DOM is ready.

## Routing

History API routing — the URL pathname encodes the full path (`/category/page`, `/category/folder/page`). An in-page anchor is carried in the hash (`/math/algebra#section`).

On every `popstate` or programmatic `navigate()`, the router strips the leading slash, validates the path against `/^[a-zA-Z0-9-_/]*$/` to block directory traversal, then calls `onRouteChange` in `app.js` to load content. The server serves `index.html` for all routes; the browser never does a hard navigation.

## Content Loading

`ContentLoader` maps a route to a markdown file path, fetches it, rewrites relative links to root-relative paths, then hands the parsed HTML to Alpine. The mapping is straightforward: URL segments become file path segments (`math/precalculus/summary` → `/math/precalculus/summary.md`). Folder routes load the folder's `README.md` automatically.

**Cache:** parsed HTML is stored in a `Map` keyed by file path. Repeat visits skip the fetch and re-parse entirely — no expiry, lives for the session.

**Same-page navigation:** clicking a search result that points to an anchor on the currently loaded page skips the content reload entirely and scrolls directly to the element.

## Search

Full-text search with a flat in-memory index built at startup.

**Indexing:** all markdown files are fetched in parallel (`Promise.all`). Each file is scanned line-by-line and split on headers (`#`, `##`, `###`) into sections. Each section becomes an index entry: `{ title, category, page, link, plainText }`. The link includes the in-page anchor so clicking a result scrolls directly to the matched section.

**Query:** linear scan over the index — `filter` by substring match on `title` and `plainText`. Results are capped at 10, queries are debounced at 300 ms, max query length is 100 characters.

**Scalability note:** at 200+ pages, index build time and memory cost become the bottleneck. The right fix is to move index generation to build time and serve `dist/search-index.json` — the same move already made for navigation data.

## Rendering Pipeline

After Alpine flushes new content to the DOM, a `MutationObserver` fires and runs post-render steps in order:

1. Assign slug-based IDs to `h2`/`h3` headings (`utils.js`)
2. Build the TOC and start an `IntersectionObserver` for active-section tracking
3. Detect timeline lists (year-pattern heuristic or explicit `timeline` marker)
4. KaTeX auto-render for `$...$` and `$$...$$` (guarded by `marked-extensions.js` upstream)
5. Highlight.js on `pre code` blocks — wraps each block in a toolbar with a language selector dropdown and a copy button
6. Scroll to anchor (if present) or reset to top

`MutationObserver` replaced a fixed `setTimeout` — it fires exactly when Alpine is done, not after a fixed guess.
